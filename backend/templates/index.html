AirPulse 2.0 DBMS Design Slides

1. Hybrid SQL/NoSQL Architecture

Evaluators expect advanced SQL queries. To support both, AirPulse 2.0 uses a dual-db engine: MongoDB Atlas serves the live WebSockets client, while a synchronized SQLite database powers the SQL playground. This allows you to demonstrate SELECT, JOIN, GROUP BY, and nested queries without complex installations.

2. MongoDB Collections Schema Design

  • stations: GeoJSON geometry type `Point` for mapping capabilities.
  • pollution_records: Chronological indices tracking concentrations of PM2.5, PM10, CO, NO2, SO2, O3.
  • system_alerts: Active threat events generated automatically on write actions.
  • audit_logs: Triggers log before/after JSON states of modified documents.

DBMS Evaluator Quick Viva Prep:

Q: How do database triggers operate in your NoSQL design? A: In MongoDB, triggers are emulated at the app layer via PyMongo middleware. Inserting a pollution document automatically calculates sub-indices, writes a security alert if thresholds are exceeded, and logs a JSON snapshot to the audit logs collection.
Q: Why use GeoJSON indexes in MongoDB? A: Creating a `2dsphere` index on `stations.location` allows querying nearest stations using `$near` or `$geoWithin` operators. This enables spatial search queries on our dashboard.
