Project Abstract & Problem Statement

Project Abstract

AirPulse is a modern environmental analytics platform designed to solve urban air quality monitoring challenges using advanced Database Management System (DBMS) concepts. The system organizes air quality metrics ($PM_{2.5}, PM_{10}, CO, NO_2, SO_2, O_3$) captured from geographical monitoring stations, computes the standardized Air Quality Index (AQI), provides context-aware health advisories, and raises automated system alerts when safe pollution limits are breached.

Rather than treating the database as a passive storage engine, AirPulse showcases dynamic DBMS functions like trigger-based auditing, conditional logging, custom views, indexing, and transaction-wrapped CRUD actions. A real-time database activity log feed connects the dashboard interface with the SQLite/MySQL engine, making every insertion, update, and deletion visually manifest as dynamic events.

The Problem Statement

Academic Disconnect: Traditional college DBMS projects are often back-end operations with dry command-line text logs or simple spreadsheet-like forms. Evaluators cannot see the interactive nature of active databases (triggers, views, constraints, transactions) working in harmony with user-facing dashboards.

The AirPulse Solution: AirPulse integrates a premium graphical monitoring analytics dashboard with a fully reactive DBMS playground. By creating automated database listeners (triggers), the system logs every database change into a visible audit table, demonstrates instantaneous safety alerts, and updates live historical trends. This bridges the gap between database theory and visual software architecture.

Entities & Attributes

  • Stations (Weak Entity-Parent): Represents active sensors.
    • StationID (PK) - Uniquely identifies stations.
    • Name (Unique), City, Latitude, Longitude, Status, EstablishedDate.
  • PollutionRecords (Weak Entity-Child): Environmental metrics.
    • RecordID (PK), StationID (FK), Timestamp.
    • Pollutants: PM25, PM10, CO, NO2, SO2, O3.
    • Derived metrics: AQI, AQI_Category (FK).
  • HealthAdvisories (Lookup Entity):
    • CategoryName (PK) - e.g., 'Good', 'Severe'.
    • AQILower, AQIUpper, GeneralAdvisory, ChildElderlyAdvisory, ActionTip, ColorCode.
  • SystemAlerts (Event Entity):
    • AlertID (PK), StationID (FK), RecordID (FK), Pollutant, ObservedValue, ThresholdValue, AlertTimestamp, Status.

Normalization Analysis (Up to 3NF)

Normalization is the process of structuring relational database columns to minimize redundancy and dependency. Here is how AirPulse satisfies normalization principles:

1st Normal Form (1NF)

Rule: All attributes must contain atomic (indivisible) values. There are no repeating groups or multi-valued attributes.

Satisfied: All concentrations, station details, coordinates, and timestamp values are scalar and atomic.

2nd Normal Form (2NF)

Rule: Must be in 1NF, and all non-key attributes must be fully functionally dependent on the entire primary key. (No partial dependencies).

Satisfied: Since all our primary keys are single-column identifiers (StationID, `RecordID`, `AlertID`), partial key dependencies are mathematically impossible.

3rd Normal Form (3NF)

Rule: Must be in 2NF, and no non-key attribute should transitively depend on the primary key (No transitive dependencies: $A \rightarrow B \rightarrow C$).

Satisfied: In `PollutionRecords`, values like `PM25` or `CO` depend directly on the `RecordID`. The advisory details (e.g. `GeneralAdvisory`, `ColorCode`) depend on `AQI_Category`, which in turn depends on `AQI`. Storing health advisories in a separate `HealthAdvisories` table with `CategoryName` as the primary key completely eliminates transitive dependencies from `PollutionRecords` to keep it strictly in 3NF.

Triggers, Views, and Advanced DBMS Concepts

1. Database Triggers (Automated Actions)

Triggers are precompiled SQL blocks that fire automatically on specific modifications (`INSERT`, `UPDATE`, `DELETE`). AirPulse implements three triggers:

  • Audit Tracker Trigger: Records old and new states in the `AuditLog` table on any pollution record changes.
  • System Alert Generator: Checks incoming pollutant levels on insert. If $PM_{2.5} > 150.0$, $SO_2 > 75.0$, or $AQI > 200$, it inserts an active threat row into the `SystemAlerts` table.
  • Station Audit Trigger: Logs registration details of new sensors automatically.

2. Database Views (Simplified Querying)

A View is a virtual table representing the result of a saved query. We define `DetailedPollutionDashboard` to join three tables: PollutionRecords, Stations, and HealthAdvisories.

This hides query complexity from the application layer. Instead of writing heavy joins in Flask, the front-end requests SELECT * FROM DetailedPollutionDashboard. This improves modularity and security.

3. Indexes (Performance Tuning)

We declare indexes on PollutionRecords(StationID) and PollutionRecords(Timestamp). This reduces the search complexity from $O(N)$ (table scan) to $O(\log N)$ (B-Tree lookup) during dashboard trend rendering.

DBMS Viva Voce Preparation (Q&A)

Q1: What is the benefit of using Foreign Key Constraints like "ON DELETE CASCADE"?

Answer: It enforces referential integrity. In AirPulse, if an admin deletes a Station, all child PollutionRecords and SystemAlerts pointing to that StationID are automatically purged by the database engine. This prevents "orphan records" and maintains data consistency.

Q2: Why did you compute the AQI category via python/triggers instead of storing it statically?

Answer: Storing computed categories statically creates transitive dependencies (AQI determines Category), which violates 3NF. In our system, the database references a lookup table HealthAdvisories via a Foreign Key, and we join on it, maintaining strict normalization while maximizing speed.

Q3: How do database triggers help in security auditing?

Answer: Triggers run directly inside the database kernel. Even if an attacker bypasses the Flask application and modifies the tables directly via an SQL client (like phpMyAdmin), the database triggers will still fire and log their username, timestamp, and query details into the AuditLog. This provides tamper-proof auditing.

Q4: What is the difference between a Database View and a Database Table?

Answer: A Table stores physical data on disk. A View is a virtual table containing a saved SQL query. It does not occupy storage; rather, it dynamically pulls data from underlying tables whenever it is queried. Views simplify frontend SQL queries and restrict user access to sensitive columns.

Q5: Explain how database transactions ensure reliability in CRUD operations.

Answer: Transactions follow the ACID properties (Atomicity, Consistency, Isolation, Durability). In our system, if an insert query fails halfway (e.g. fails a check constraint or foreign key check), the database rolls back all partial operations to ensure the database never reaches an inconsistent state.
